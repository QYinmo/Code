## 一、必须修复：UART 等待期间可能越过帧截止时间

当前 `UartTxStateMachine::process()` 只在函数开始时检查：

```cpp
if (pending_valid_ && now - pending_created_at_ > frame_deadline_)
```

之后如果 `writeSome()` 返回 `would_block`，代码会调用 `waitWritable(poll_timeout_ms_)`，随后可能直接再次调用 `writeSome()`。这里仍使用调用者传入的旧 `now`，没有在等待结束后重新读取 `steady_clock` 并检查截止时间。

因此，如果 `waitWritable()` 消耗了剩余时间，已经过期的旧非零帧仍可能在同一次 `process()` 调用中继续发送完成。这不符合“pending 帧具有严格发送截止时间”的安全要求。

请整改为：

1. `waitWritable()` 返回后、任何后续写入前，必须使用真实的单调时钟重新检查截止时间。
2. 帧一旦超过截止时间，不得再写入该旧非零帧的剩余字节。
3. 超时后继续执行现有的通信故障和重同步策略：
   - 统计 `deadline_timeouts`；
   - 丢弃本机尚未发出的输出；
   - 清理 pending 状态；
   - 锁存通信故障；
   - 恢复后首先发送完整零速度 STOP；
   - 不补发旧非零目标。
4. 已经到达线路的半帧无法撤回，因此不得删除 README 中对 MSPM0“滑动搜索 `5A A5`、固定 10 字节、验证 mode 和 CRC”的重同步要求。
5. 不要因为修复截止时间而在控制线程中加入无界阻塞。

### 时钟可测试性要求

不要写依赖真实 `sleep`、容易偶发失败的单元测试。建议为 `UartTxStateMachine` 注入可控单调时钟或“当前时间提供器”，也可以采用其他确定性的设计，但必须让 fake writer 能在 `waitWritable()` 期间推进模拟时间。

如果修改 `process()` 的接口或 writer 接口，请保持生产实现使用 `std::chrono::steady_clock`，并同步更新所有调用点。

## 二、必须补充：UART 截止时间边界测试

当前 `tests/test_uart_tx.cpp` 已覆盖完整写、1+9 部分写、多次 `EAGAIN`、下一次调用发现截止超时和恢复后优先 STOP，但没有覆盖“单次 `process()` 内，poll 等待期间跨过截止时间”。

请增加确定性测试，至少覆盖：

1. 非零 RATE_CONTROL 帧先成功写入 1 字节。
2. 下一次写入返回 `EAGAIN`。

4. 状态机不得再写剩余 9 字节。
5. 返回事件必须是 `deadline_timeout`，并增加对应统计。
6. 恢复后发送的第一完整帧必须是零速度 STOP。
7. 旧非零帧不得在恢复后继续补发。

同时保留现有 UART 测试，不得用新测试替换原有覆盖。

## 三、必须补强：UART poll 超时与帧截止时间的配置约束

当前 `validateConfig()` 只检查 `uart_poll_timeout_ms <= 10`，没有验证 poll 等待必须小于 `uart_frame_deadline_s`。

请增加合理的配置约束：

- 将 `uart_poll_timeout_ms` 换算为秒后，必须严格小于 `uart_frame_deadline_s`；
- 应保留足够余量，使一次短等待后仍有机会完成 10 字节小帧；
- 不要只依赖配置约束来保证安全，运行时仍必须重新检查真实截止时间；
- 错误信息应明确指出是 UART poll 等待与帧截止时间不匹配。

请在 `tests/test_config.cpp` 增加无效配置测试，至少验证：

- poll 超时等于帧截止时间时被拒绝；
- poll 超时大于帧截止时间时被拒绝；
- 当前默认配置仍能通过验证。

## 四、必须修复：真正的校准失败不能返回成功退出码

当前 `Application::run()` 在 `calibrate()` 返回 `false` 后会进入 FAULT、发布停止/故障命令并调用 `requestStop()`，但没有可靠记录“这是程序故障”。最终可能由：

```cpp
return fatal_error_.empty() ? 0 : 2;
```

返回 `0`，导致 systemd、启动脚本或自动化程序把校准失败、SPI 连续失败等情况误认为正常成功退出。

请明确区分：

- 用户通过 `SIGINT`、`SIGTERM`、`quit` 或正常停止流程退出：可以返回 `0`；
- BMI088 初始化失败、UART 打开失败、真实校准失败、校准期间达到 SPI 连续失败阈值等启动/运行故障：必须返回非零；
- 校准期间用户主动发送 SIGINT/SIGTERM，不应被误报为“校准故障”；
- 无论返回码是什么，都必须保留发送多个 STOP 包、关闭 UART/SPI 和线程回收的安全退出流程。

请优先使用语义明确的退出原因或退出状态，不要仅靠模糊字符串判断。

### 退出码测试要求

请尽量把“校准结果/退出原因到进程返回码”的决策提取为可单元测试逻辑，避免测试依赖真实 BMI088。

至少验证：

1. 正常用户停止返回 `0`。
2. 信号请求的正常安全退出返回 `0`。
3. 校准算法或 SPI 失败导致的校准失败返回非零。
4. BMI088/UART 初始化失败仍返回非零。

如果完整 `Application` 难以注入假 BMI088，可以测试提取后的退出状态决策，但必须说明哪些集成行为仍需实机验证。

## 五、重新核对第一次整改不得回归

完成上述修改后，再次检查以下行为仍成立：

- 默认 `auto_run=false`。
- BMI088 初始化和校准成功后保持 `STOP_READY`。
- 只有人工 `run`、明确的 `--auto-run` 或语义明确的测试模式才允许 RATE_CONTROL。
- `run` 前检查姿态有效、IMU 新鲜、UART 可用和目标有限。
- STOP、故障和运行模式切换会清空 PID 积分、历史和目标斜坡。

- Pi 5 UART 文档仍要求用实机命令确认 GPIO14/GPIO15 对应设备节点。
- `原始/` 中参考文件没有被修改。
