## 一、当前已经正确的功能不得回归

整改时必须保留：

- BMI088 加速度计使用 `/dev/spidev0.0`，陀螺仪使用 `/dev/spidev0.1`。
- 加速度计 SPI 读取包含额外 dummy byte，陀螺仪没有。
- 每次寄存器读取使用单次 `SPI_IOC_MESSAGE(1)`。
- ±6 g 和 ±500 °/s 换算正确。
- 校准只把陀螺仪均值作为零偏，不从运行加速度中删除重力。
- 默认 `auto_run=false`。
- 普通模式校准完成后保持 `STOP_READY`，人工输入 `run` 后才进入角速度控制。
- UART 10 字节固定包、小端字段和 CRC-16/MODBUS。
- UART 部分写、EAGAIN、严格帧截止时间和恢复后优先 STOP。
- Mahony 加速度可信度门控、积分衰减和积分限幅。
- STOP、FAULT 和模式切换清空 PID 状态。
- SIGINT/SIGTERM 处理器只执行异步信号安全操作。
- 退出时发送多个零速度 STOP 包并关闭设备。
- `mode=2` 姿态串口模式不运行 PID、不发送角速度控制帧。
- 姿态串口模式只允许 `status` 和 `quit`，禁止 `run`、`stop`、`target` 和 `reset-fault`。
- `原始/` 中的参考资料保持不变。

## 二、必须完成：MSPM0 流式 UART 接收与重同步状态机

当前 `mspm0_protocol/gimbal_protocol.c` 主要负责验证和解析一个已经切分好的 10 字节包，但真实 UART 是连续字节流，不能假设每次 DMA、FIFO 或中断回调都恰好收到完整的一包。

请在 `mspm0_protocol/` 中增加可移植的纯 C 流式解析模块。不能依赖树莓派、Linux 或具体 MSPM0 SDK，使它能够在 MSPM0 工程和主机单元测试中复用。

必须支持：

1. 逐字节或任意长度分块输入。
2. 滑动搜索双字节包头 `5A A5`。
3. 找到包头后收集固定 10 字节候选帧。
4. 检查 mode 范围和 CRC。
5. 候选帧无效时只前移一个字节继续搜索，不能直接清空全部输入。
6. 能处理以下输入：
   - 一次收到完整 10 字节；
   - 每次只收到 1 字节；
   - 1+9、2+8、任意分块；
   - 包头前存在噪声；
   - 两个正确包连续粘连；
   - CRC 错误包后紧跟正确包；
   - 缺失字节导致错位；
   - 数据中间出现类似 `5A A5` 的字节；
   - sequence 自然从 `0xFF` 回绕到 `0x00`。
7. 解析器不得动态分配内存。
8. 所有数组访问必须有边界保护。
9. 提供明确统计：
   - 输入字节数；
   - 成功帧数；
   - CRC 或候选帧错误数；
   - 丢弃或重同步字节数；
   - sequence 跳变次数。
10. sequence 跳变只用于诊断，不得单独触发电机故障，因为它可能来自本机发送失败、链路丢包或自然回绕。

建议新增文件，也可以采用同等清晰的命名：

```text
mspm0_protocol/gimbal_stream_parser.h
mspm0_protocol/gimbal_stream_parser.c
tests/test_mspm0_stream_parser.cpp
```

请保持现有 `GimbalProtocol_IsValid()` 和 `GimbalProtocol_Parse()` 的兼容性。

## 三、必须提供：MSPM0 通信安全监控的可移植逻辑

请为 MSPM0 端提供不依赖具体定时器驱动的安全监控状态逻辑，由调用者传入单调时间或周期计数。

必须表达并测试：

- 收到合法 `RATE_CONTROL` 后更新目标和最后接收时间。
- 收到 `STOP` 后立即把目标归零。
- 收到 `FAULT` 或未知 mode 后立即进入故障停止。
- 收到 `ATTITUDE` 时不得把第 5～8 字节误当电机角速度。
- 超过约 20～30 ms 没有收到新的有效角速度目标时，目标开始降速或归零。
- 超过约 100 ms 没有收到合法控制帧时，进入通信故障。
- 恢复通信后不能立刻恢复旧的非零目标，必须先收到明确的新控制命令，并遵守当前安全策略。
- mode 2 姿态包不能喂给电机速度闭环。

数值门限必须通过结构体或宏配置，并验证：

```text
target_timeout < communication_fault_timeout
```

树莓派不能替代 MSPM0 的机械限位、过流、过温、急停和最终电机保护。对于依赖具体电机驱动和硬件的部分，可以提供明确的接口和集成说明，但不能谎称已经完成真实硬件验证。

## 四、必须补齐：姿态串口模式的应用级测试

当前 `tests/test_attitude_uart.cpp` 主要验证 mode 2 的角度编码和 CRC，还不足以证明完整应用行为。

请将姿态串口模式的决策逻辑提取成可测试组件，或用可注入 fake IMU、fake writer、fake clock 的方式增加确定性测试。不得依赖真实 BMI088、UART 或真实时间睡眠。

至少验证：

1. 校准未完成时不发送姿态帧。
2. 姿态尚未有效时不发送姿态帧。
3. 首次有效姿态后发送 `mode=2`。
4. mode 2 的第 5～6 字节是 yaw，单位 `0.01°`。
5. mode 2 的第 7～8 字节是 pitch，单位 `0.01°`。
6. 连续 yaw 正确折回 `[-180°, 180°]`。
7. 姿态数据超过 `command_timeout_s` 后停止发送旧姿态。
8. 姿态滤波进入降级或姿态无效时不继续发送旧姿态。
9. 姿态模式不能进入 `RUNNING`。
10. 姿态模式不产生 `RATE_CONTROL` 帧。
11. `run`、`stop`、`target`、`reset-fault` 在姿态模式被拒绝。
12. `status` 和 `quit` 仍可使用。
13. UART 系统调用错误或帧截止超时会记录失败退出状态并安全退出。

如果完整 `Application` 难以测试，请提取“当前运行选项、姿态有效性、安全状态到待发送包”的纯决策逻辑。测试必须覆盖真实生产代码，不能重新在测试中复制一份等价逻辑。

## 五、必须改进：协议字段命名不能混淆角度和角速度

当前树莓派端 `GimbalCommand` 使用类似：

```cpp
yaw_rate_dps
pitch_rate_dps
```

但在 `ATTITUDE` 模式中，这两个字段实际表示姿态角。

当前 MSPM0 端 `GimbalDecodedCommand` 使用类似：

```c
yaw_rate_cdeg_s
pitch_rate_cdeg_s
```

但在 mode 2 中，它们实际表示 `0.01°` 的角度。

这种复用容易让后续接收代码把姿态角误当角速度。

请采用语义清晰且不会破坏线协议的设计，例如：

- 使用中性原始字段名：

```text
yaw_value
pitch_value
```

- 或使用明确的 payload/union/accessor：

```text
rate_control.yaw_cdeg_s
attitude.yaw_cdeg
```

要求：

1. 线上的 10 字节协议不能改变。
2. 保持小端和 CRC 完全兼容。
3. 树莓派与 MSPM0 两端命名必须一致或容易对应。
4. 不允许调用者在不检查 mode 的情况下直接把字段当作角速度。
5. README、注释和测试同步更新。
6. 如需保留旧接口兼容性，必须清楚标记弃用或提供安全转换函数。

建议提供按 mode 解读的函数，例如：

```text
GimbalProtocol_GetRateControl(...)
GimbalProtocol_GetAttitude(...)
```

这些函数在 mode 不匹配时必须返回失败。

## 六、必须明确并测试：姿态模式退出时的 STOP 包行为

当前程序在姿态串口模式运行时持续发送 mode 2，但 `quit`、SIGINT 或 SIGTERM 退出时，公共的 `sendShutdownPackets()` 会发送多个 mode 0 STOP 包。

从电机安全角度，退出时发送 STOP 是合理的；但纯姿态接收器会在连续 mode 2 后看到若干 mode 0，因此协议和测试必须明确这个行为。

请采用并固定一种策略：

### 推荐策略

姿态串口模式退出时仍发送 5 个 mode 0 STOP 包，原因是同一串口可能连接 MSPM0，安全停止优先。

如果采用推荐策略，必须：

- README 明确说明姿态流退出时会出现 mode 0；
- 接收端按 mode 分发，不能把 mode 0 当作姿态值；
- 增加测试：姿态模式正常运行发送 mode 2，退出阶段发送 5 个零值 STOP；
- STOP 的 yaw/pitch 字段必须为 0；
- 不得在退出阶段发送旧姿态或 RATE_CONTROL。

如果选择其他策略，必须说明为什么不会降低电机安全性，并提供相应测试。不要只靠注释决定行为。
